## Item to ship

Cut 08 had a "Previous beat" / "Advance beat" button pair next to the
command line. Our app auto-advances phases via the scenario clock —
which means a speaker can't pause on a beat for Q&A or rewind during
the demo. This is a real LIVE DEMO CONTROL gap. Tier 1 priority.

The replay controls render as a small button group inline with the
command line at the bottom. Layout:

```
> /help, /reset, /save-rule, /event 1|2          [⏮ prev] [▶ pause] [⏭ next]
```

Three buttons:
- **Previous beat** (⏮) — jumps phase from N to N-1, or no-op at P1
- **Play/pause** (▶/⏸) — toggles scenario clock between running and paused
- **Next beat** (⏭) — jumps phase from N to N+1, or no-op at P6

Each button POSTs to `/scenario/control` with the appropriate command
(`previous_phase` / `pause` / `play` / `next_phase`). If a command
isn't supported by the existing route, fall back to client-side state:
manually compute the next phase and dispatch a custom event
`liminal:phase-control` that the scenario state machine can listen on.

**Visual register:** button-group register, mono caption, decision-blue
on hover, semi-transparent background. Pause button shows the OPPOSITE
icon when active (▶ when paused, ⏸ when playing).

**Behavior:**
- Default state: playing (matches current behavior)
- After clicking pause: clock stops; phase doesn't auto-advance until
  play is clicked again
- Prev/next work whether playing or paused — they jump phase directly
