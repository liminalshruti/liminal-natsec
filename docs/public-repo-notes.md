# Public Repo Notes

A reality contract for anyone reading this repo cold — a judge after the demo, a follow-on principal, a mentor, a future contributor. **Liminal Custody is a hackathon submission.** It is a working technical demo, not a deployed system. This page makes the boundary between "real and validated," "fixture / cached," and "intentionally not claimed" explicit.

For the product positioning, read [`liminal-custody-onepager.md`](liminal-custody-onepager.md) first.

**Demo vs Q&A modes** (per `docs/integration-state.md` §6):

- **Demo mode (default):** fixtures → guard → UI. Deterministic timing. The structural guard fires on the same evidence in the same way every time.
- **Q&A mode (env flag flip):** operator question → AIP Logic → guard → UI. The judge cannot visually distinguish the two. Same guard, same UI component.

Both modes go through the same `server/src/specialists/guard.ts`. The refusal beat is enforced regardless of source.

## What is intentionally not claimed

This list is as load-bearing as the feature list. We do not stretch the demo past what it can deliver.

- **Not a live AIS feed during the demo.** Real AIS is in the cached data; the timed pitch runs from fixtures.
- **No live LLM on the critical path.** AIP Logic is wired and accessible (developer tier), but the pitch path runs from fixtures. AIP is the Q&A fallback.
- **No hostile-intent inference.** The Intent specialist refuses by default unless the structural guard's `INTENT_INDICATOR` requirement and citation minimums are met.
- **No automated escalation.** The system recommends collection actions; humans choose.
- **No command-action authority.** We protect the layer *before* command. Liminal Custody does not issue orders.
- **Not a Maven replacement.** Maven is the Pentagon's program of record for CJADC2. We slot one layer earlier (substrate → custody → command). See `docs/maven-analysis.md`.
- **No real DoD operational data.** Hormuz AIS replay is constructed from public sources for illustrative purposes.
- **No live targeting.**
- **No live attribution.** Identity-churn cases preserve hypotheses; they do not declare which MMSI is "true."
- **Not production-ready.** This is a hackathon demo. Pilot path is described in `docs/v4-judge-calibrated-demo.md`.

## Why this matters for the pitch

The structural guard at `server/src/specialists/guard.ts` runs **server-side, on every specialist output, regardless of source**. Whether the input is a cached fixture or a live AIP Logic call, the same 7+ layered checks (citation minimum, evidence-type requirements, posterior thresholds, Shodan-only restriction on vessel-behavior claims, etc.) decide whether an output is allowed to render or is forced to `verdict: refused`.

That means the headline Round 1 beat — *"Intent refused because Signal Integrity contested"* — is the same beat in demo mode and Q&A mode. The guard is the invariant. Fixtures are an optimization for deterministic timing during the timed pitch; they are not a workaround for missing functionality.

This is why the team is comfortable saying *"refusal is structurally enforced, not requested"* in pitch.

## Foundry / AIP integration status

- AIP Logic developer access is provisioned for the team. Specialist routes are wired in `server/src/specialists/aip.ts` and dispatched through `server/src/specialists/registry.ts`.
- Live AIP path: operator question → registry → AIP Logic → response → `applyGuard()` → render.
- Demo critical path: fixture lookup → `applyGuard()` → render.
- Foundry ontology creation spec lives at `foundry/seaforge-ontology-spec.md` (note: legacy product name in filename; canonical product name is "Liminal Custody"). Identifier checklist at `foundry/identifiers.md`.
- Live Foundry RIDs, AIP query API names, OAuth client IDs, dataset RIDs are all configured via `config.ini` (gitignored). The `config.example.ini` template enumerates every config key with empty placeholders.
